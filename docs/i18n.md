# Internationalization (i18n)

Shield's web UI uses [vue-i18n](https://vue-i18n.intlify.dev/) v11 (composition API, legacy mode off). The operator picks a language in the top bar; the choice is persisted in `localStorage` under `shield.locale` and survives reload. No rebuild required to switch languages.

## Supported locales

| Code | Language        | Status                          |
|------|-----------------|---------------------------------|
| `en` | English         | Reference — every key defined   |
| `nl` | Nederlands      | Starter coverage (auth + nav + severity) |
| `de` | Deutsch         | Stub — contributions welcome    |
| `fr` | Français        | Stub — contributions welcome    |
| `es` | Español         | Stub — contributions welcome    |

Missing keys in non-English locales fall back to English at runtime.

## File layout

```
src/Shield.Web/src/i18n/
  index.ts                 # createI18n() + setLocale()
  locales/
    en.json                # canonical message shape (MessageSchema)
    nl.json
    de.json
    fr.json
    es.json
```

`en.json` is preloaded into the main bundle. Other locales are dynamically imported when first selected, so they ship as separate chunks and don't bloat the initial page weight.

## Adding a new locale

1. Pick the ISO 639-1 two-letter code (e.g. `it` for Italian).
2. Add the code to `SUPPORTED_LOCALES` in `src/i18n/index.ts`, and add label/flag entries to `LOCALE_LABELS` / `LOCALE_FLAGS`.
3. Create `src/i18n/locales/<code>.json`. Use `en.json` as the schema reference.
4. Translate values, leaving `{placeholders}` untouched (vue-i18n interpolates them at render time).
5. Rebuild: `npm run build`.

That's it — the language picker in the top bar will pick the new locale up automatically.

## Using translations in components

**Templates** (global `$t` injection is on):
```vue
<span>{{ $t('nav.dashboard') }}</span>
```

**Script (composition API)**:
```ts
import { useI18n } from 'vue-i18n'
const { t } = useI18n()
const message = t('error.signin_failed')
const greeting = t('screen.account.signed_in_as', { user: 'arc' })
```

## Conventions

Keys are **domain-grouped + snake_case + role/action**. Leaf names describe the role of the string, not the English text — the same key keeps working if the visible copy changes.

- **Top-level domains** describe the kind of thing, not the page: `action.*` (verbs), `state.*` (verb-ing states), `field.*` (form labels), `nav.*` (nav labels), `screen.*` (page-specific copy), `severity.*`, `banner.*`, `toast.*`, `error.*`.
- **Screen sub-grouping** one level deep: `screen.signin.title`, `screen.register.submit_btn`, `screen.account.signed_in_as`.
- **Snake_case leaves with role suffixes**: `submit_btn`, `with_provider`, `closed_body`, `confirm_password`, `password_updated`. Never `signInBtn`, never `signInButton`.
- **Placeholders** use `{name}` syntax. Never translate the placeholder name.
- **Reuse** before adding: `action.save`, `state.loading`, `field.username` already exist.
- **English source of truth**: when you add a UI string, add the English key in `en.json` in the same commit; other locales can lag.

### Examples

| Bad (English-derived)        | Good (role/action)                   |
|------------------------------|--------------------------------------|
| `auth.signInBtn`             | `screen.signin.submit_btn`           |
| `auth.signInWith`            | `screen.signin.with_provider`        |
| `common.save`                | `action.save`                        |
| `common.loading`             | `state.loading`                      |
| `auth.username`              | `field.username`                     |
| `singleUserBanner`           | `banner.single_user_mode`            |
| `nav.signOut`                | `nav.sign_out`                       |

## Backend error codes (follow-up)

Today the API returns human-readable English error strings in response bodies (e.g. `{ "error": "Registration is closed." }`). The UI surfaces these directly, which means they can't be translated without intercepting and string-matching.

The proper fix is for the API to return stable machine-readable error codes alongside the human-readable message:

```json
{ "code": "registration_closed", "error": "Registration is closed." }
```

The UI would then map `code` to an i18n key like `errors.registrationClosed` and fall back to `error` only when the code is unknown. This is a separate backend work item — track it in the v0.2 milestone.

The `Accept-Language` header is not currently consulted by the API; backend log messages stay in English by design (operator-facing, not user-facing).
